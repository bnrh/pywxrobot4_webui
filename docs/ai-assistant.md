# 智能插件

## 1. 能力定位

“智能插件”页面对应的核心实现位于 [ai_assistant.py](../ai_assistant.py) 和 [server.py](../server.py)。

它的目标不是单纯聊天，而是让模型在当前微信上下文里调用 wxrobot_api 工具完成实际操作，例如：

- 查询登录账号、群聊、好友、群成员。
- 发送文本、图片、文件、视频、GIF。
- 获取聊天记录、标签、联系人详情。
- 进行群成员交集统计、群好友交集统计等聚合查询。

## 2. 当前支持的模型厂商

当前内置厂商目录来自 [ai_assistant.py](../ai_assistant.py) 的 `PROVIDER_CATALOG`：

| 厂商键 | 页面显示 | 说明 |
| --- | --- | --- |
| `zhipu` | 智谱 | GLM 系列 |
| `deepseek` | DeepSeek | DeepSeek Chat |
| `minimax` | MiniMax | MiniMax 文本模型 |
| `openai` | 通用 OpenAI | 兼容 OpenAI 协议的任意网关 |

每个厂商都可以在页面里维护多个配置项，并单独启用或停用。

## 3. 工具执行模型

当前智能插件不是简单拼装 HTTP 接口列表，而是两层工具面合并：

1. 本地增强工具：例如 `count_shared_room_members`、`list_shared_room_members`、`count_room_friend_members`、`list_room_friend_members`。
2. wxrobot_api MCP 工具：从 `wxrobot_api/api/mcp_server.py` 提取工具定义，并通过 `/mcp` 能力执行。

这意味着模型能直接看到一份更稳定、更适合工具调用的 schema，而不是依赖页面层自己拼接接口说明。

## 4. 为什么这里强调聚合工具

当前智能插件已经内置了一套工具路由规则，用来避免模型把超大群聊或超大好友列表完整读出来再自己算。

例如：

- 问两个群是否有重复成员时，优先调用 `count_shared_room_members` 或 `list_shared_room_members`。
- 问某个群里有多少成员已经是你的好友时，优先调用 `count_room_friend_members` 或 `list_room_friend_members`。
- 遇到“今天、昨天、最近几小时”等相对时间时，优先调用 `current_datetime`。

这套路由规则的核心目的是：

1. 降低 token 消耗。
2. 减少模型拿不到完整列表时的截断问题。
3. 让结果更可预测。

## 5. 对话与任务模型

智能插件当前采用“对话 + 后台任务”的组合模式。

### 对话

每个对话会保存：

- 标题
- 创建时间
- 更新时间
- 消息列表

对话历史会持久化到 SQLite，并且支持：

- 新建对话
- 切换对话
- 清空对话

### 后台任务

每次实际提交给模型的请求，会生成一个异步任务，用于处理：

- 流式进度更新
- 工具调用轨迹
- 失败状态
- 手动停止

这样做的好处是：复杂工具调用不会阻塞整个页面交互。

## 6. 页面操作建议

推荐的使用顺序：

1. 先在“配置”里保存至少一个可用的厂商配置。
2. 选择厂商和模型。
3. 先用只读查询验证工具链是否可用。
4. 再尝试发送消息、打标签、拉群等写操作。

对于写操作，建议显式给出：

- wxid
- roomid
- wxpid
- 文件路径
- 要发送的具体内容

这样能降低模型补全参数时的歧义。

## 7. 安全与边界

当前默认系统提示词强调了两个原则：

1. 写操作前应拿到足够参数，不要臆造 wxid、roomid、wxpid 或路径。
2. 只读查询优先通过工具获取事实，再给出结论。

因此，如果模型缺参数，它更倾向先向用户追问，而不是直接猜。

## 8. 常见问题

### 模型为什么不先列全量群成员再分析

因为当前系统已经对“重复成员”“群内好友成员”等问题增加了聚合工具优先策略，目的就是避免大列表截断。

### 为什么有的工具看起来不是页面里手写定义的

因为工具定义会从 wxrobot_api MCP 服务中提取并合并进来，这也是当前智能插件和旧文档里“HTTP 接口拼装”思路最大的区别之一。

### 对话能不能中断

可以。当前页面支持停止仍处于 `queued`、`running`、`stopping` 状态的对话任务。
